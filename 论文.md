# 中文摘要及关键词

随着健康消费需求持续增长，传统保健品电商平台在个体差异识别、科学选购指导和个性化服务方面仍存在不足。为提升用户选购效率与商品匹配度，本文设计并实现了一套基于用户画像的个性化健康营养商城系统。系统采用前后端分离架构，前端基于 Vue3 与 Element Plus 构建交互页面，后端基于 Spring Boot 提供 REST 接口服务，结合 MySQL 完成业务数据持久化，并通过 JWT 实现身份认证与权限控制。系统实现了用户注册登录、画像问卷采集、个性化商品推荐、商品浏览、购物车管理、订单提交、个人中心以及后台商品与订单管理等功能。在推荐方法上，本文采用规则加权评分模型，将健康目标匹配、预算匹配、目标人群匹配和行为增强项进行综合计算，并以匹配度形式展示推荐结果，从而增强推荐过程的可解释性。测试结果表明，系统主要业务流程运行稳定，能够满足普通用户个性化选购和管理员后台运营的基本需求，具有较好的实用性与扩展性。

# 英文摘要及关键词

With the continuous growth of health consumption, traditional health-product e-commerce platforms still show clear limitations in user differentiation, scientific purchasing guidance, and personalized service. To improve shopping efficiency and product relevance, this thesis designs and implements a personalized health nutrition mall system based on user profiling. The system adopts a front-end and back-end separated architecture. The front end is developed with Vue3 and Element Plus for interactive pages, while the back end is built with Spring Boot to provide RESTful services. MySQL is used for business data persistence, and JWT is adopted for authentication and authorization. The system supports user registration and login, profile survey collection, personalized product recommendation, product browsing, shopping cart management, order submission, user center, and admin-side product and order management. For recommendation, a rule-based weighted scoring model is designed by integrating health-goal matching, budget matching, target-audience matching, and behavior boost. The recommendation results are displayed with matching scores to improve interpretability. Test results show that the major business processes run stably and can satisfy the basic needs of personalized shopping for users and operation management for administrators. The system therefore has good practicality and provides a feasible basis for further expansion in intelligent recommendation and health-service applications.

# 第一章 绪论

## 1.1 研究背景及意义

### 1.1.1 研究背景

保健品本质上是以改善人体机能、补充营养缺口和辅助健康管理为目的的功能性消费产品。近年来，受消费升级、亚健康人群扩大、生活方式变化以及健康意识提升等因素共同驱动，保健品市场持续扩张。相关研究显示，2020 年全球膳食补充剂市场规模达到 1562 亿美元，美国草药膳食补充剂销售额达到 112.61 亿美元并创下 17.3% 增速\[7]；我国保健食品市场规模同样保持较高增长，并已成为全球重要市场之一\[2]\[6]。

从需求结构看，消费者对保健品的诉求已由“补充营养”逐步转向“健康管理”“预防保健”和“长期养护”\[2]。同时，保健品消费呈现年轻化趋势，年轻用户更加依赖线上渠道，且对成分透明度、品牌可信度、使用体验和个性化服务提出了更高要求\[1]。

然而，现有多数保健品电商平台仍以商品展示和营销转化为核心，在科学补充建议、成分安全提示、剂量合理性分析和人群适配解释方面能力不足。面对用户营养知识不均衡、个体差异明显、组合补充复杂等现实问题，容易出现盲目购买和不合理搭配，进而带来潜在健康风险\[4]。

### 1.1.2 研究意义

基于以上背景，本研究采用前后端分离架构，利用用户画像建模、规则加权评分、数据持久化等技术进行深入探索。通过 Spring Boot 后端框架与 Vue3 前端技术的结合完成个性化健康营养商城系统的构建，使用 MySQL 数据库和 JWT 安全认证机制，对多维度的商品标签与用户画像数据进行采集、处理和分析，提高了健康消费引导的自动化和智能化水平，为个性化选购和营养补充建议提供了更为精准的技术支持。研究成果对于推动垂直电商平台向智能化、服务化、精准化方向发展具有重要的理论和实践价值，也能够为类似健康服务系统的构建提供一定参考。

## 1.2 国内外研究现状

### 1.2.1 国内研究现状

近年来，国内关于电子商务、个性化推荐以及健康消费服务的研究持续增长。尤其在“健康中国”战略和大健康产业快速发展的背景下，如何通过互联网平台实现更科学、更精准的健康商品服务，已经成为学术界和产业界共同关注的问题。现有研究多集中于推荐算法优化、电商平台服务模式创新以及垂直健康产业数字化转型等方面，但针对“用户画像 + 推荐解释 + 交易闭环”的完整系统化实现仍相对较少。

### 1.2.2 国外研究现状

随着电子商务与人工智能技术的发展，国外已有研究尝试将用户线上购买行为与健康、生活方式特征挂钩，以评估慢性疾病风险或健康状况倾向。例如，Wang 等人的研究指出，线上消费数据可以作为预测个体健康风险和生活方式倾向的可行方法\[8]。此外，ChatDiet 等研究也证明了通过大语言模型与专业营养知识库协作，可以实现对用户个性化营养与健康建议的生成\[9]。总体来看，国外研究正在向“消费行为分析 + 健康建模 + AI 决策支持”的方向发展。

然而，尽管国外在营养推荐与健康服务结合方面已有较多实践，但多数研究仍停留在饮食建议或健康预警层面，对于“保健品补充 + 个体画像 + 推荐解释 + 电商交易”的完整闭环系统报道并不多见。这也为本课题提供了进一步研究和落地实现的空间。

## 1.3 研究内容

本文研究的核心目标是开发一个集成的基于用户画像的个性化健康营养商城系统。首先，从市场痛点与健康消费实际需求出发，对系统进行需求分析并建立需求模型。在系统架构层面，采用前后端分离模式，使用 Vue3 构建用户前端，结合 Spring Boot 与 MySQL 实现业务逻辑处理与数据持久化存储，完成购物车、订单以及后台管理等核心业务功能的闭环设计。

在个性化推荐模块，设计并实现了用户画像采集模型，构建了画像数据结构与问卷交互流程；在此基础上，提出并应用了基于规则权重的推荐算法，将用户健康需求与商品属性进行匹配，从而形成具备可解释性的推荐排序结果，把抽象的用户偏好转化为直观的个性化商品展示。最后，通过功能测试与流程测试对系统进行验证，确保各功能模块的稳定性和有效性，并针对系统不足提出后续优化方向。

# 第二章 相关开发技术

## 2.1 系统技术路线

本系统采用浏览器/服务器（B/S）架构，并基于前后端分离思想进行实现。所谓前后端分离，是指将用户界面展示与交互逻辑部署在前端应用中，将业务处理、权限控制和数据访问逻辑部署在后端服务中，二者通过 HTTP 接口进行通信。该模式已成为现代 Web 应用的主流实现方式，具有结构清晰、职责明确、便于并行开发和后期维护的优点\[15]\[16]\[18]\[19]。

从系统层次上看，本系统可划分为表现层、业务层和数据层三个部分。表现层负责面向用户提供可视化界面和交互入口，完成商品信息展示、用户登录注册、问卷填写、推荐结果查看、购物车与订单页面展示等任务；业务层负责处理系统核心逻辑，包括用户身份认证、商品查询、推荐计算、订单生成与后台管理等；数据层则负责用户、商品、订单和画像等业务数据的持久化存储。通过这种分层结构，可以降低不同模块之间的耦合度，提高代码复用性和系统扩展性。

在开发流程上，本文采用“核心接口优先、页面逐步联调、业务闭环验证”的增量式开发思路。首先根据系统业务需求划定用户认证、商品管理、购物车、订单、画像与推荐等核心接口；其次完成用户端与管理端页面开发，并通过接口联调实现页面与业务逻辑对接；最后围绕“注册登录—填写画像—获取推荐—加入购物车—提交订单—后台处理订单”的主业务链路进行联测，确保系统形成完整闭环。该开发思路能够在有限时间内优先构建系统主干功能，并逐步补充细节，符合毕业设计项目迭代式实现的特点。

## 2.2 前端技术体系

前端部分是系统与用户直接交互的载体，其实现质量直接影响用户使用体验与系统展示效果。本系统前端采用 Vue3 作为核心框架，辅以 Vue Router、Axios、Element Plus 和 Vite 等技术完成页面构建、路由组织、接口通信、组件交互与工程化管理。

### 2.2.1 Web 前端基础技术

网页开发的基础由 HTML、CSS 和 JavaScript 共同构成。HTML 负责定义页面结构，是网页内容的组织基础；CSS 负责页面样式设计与布局控制，用于提升视觉表现力和界面可读性；JavaScript 则赋予页面动态交互能力，能够响应用户操作、更新页面内容并与服务器进行异步通信。三者共同构成了现代 Web 应用开发的基础环境。

在本系统中，HTML 主要用于组织页面结构，例如首页商品卡片区域、个人中心信息区、购物车列表区、订单详情区以及后台管理表格等。通过语义化标签和组件模板，可以将商品展示、表单填写和管理操作按业务逻辑清晰划分为不同区域，从而提升页面层次感与结构可读性。CSS 则主要承担页面视觉样式和布局管理工作，包括商品列表栅格布局、按钮和表单控件样式、响应式页面间距、卡片阴影、颜色层次和移动端适配等。对于商城系统而言，合理的样式设计能够提高商品信息辨识度，使用户更快获取所需内容，并减少操作负担。JavaScript 在本系统中主要用于处理用户输入、表单校验、路由跳转、购物车数量变更、页面状态切换以及与后端 API 的交互请求等。通过异步请求机制，前端能够在不刷新页面的情况下实时获取商品列表、推荐结果和订单状态，从而提升页面的流畅性与交互体验。

### 2.2.2 Vue3 框架

Vue3 是一种渐进式 JavaScript 前端框架，具有响应式数据驱动、组件化开发和生态丰富等特点，广泛应用于中小型管理系统、商城系统和单页应用开发中\[18]\[19]。Vue3 在继承 Vue2 易用性优势的同时，引入了 Composition API、性能优化机制以及更灵活的逻辑复用方式，使得复杂页面开发与模块拆分更加高效。

本系统选择 Vue3 作为前端框架，主要基于以下原因。第一，Vue3 组件化开发模式适合本系统多页面、多模块的结构特点。首页推荐区、商品详情区、购物车条目、后台订单列表等都可以拆分为独立组件，从而提高代码复用率并降低维护难度。第二，Vue3 的响应式机制可以使数据变化自动驱动页面更新，例如推荐商品列表返回后页面自动渲染、购物车数量调整后金额自动更新、后台订单状态切换后界面实时变化等，这对于提高页面交互效率具有明显优势。第三，Vue3 配合 Composition API 能够将页面中的状态、方法和副作用逻辑进行更清晰的组织，使复杂业务页面的代码结构更加规范。

在本系统实现中，Vue3 主要用于构建首页、登录注册页、用户画像问卷页、商品详情页、购物车页、订单页、个人中心页以及后台管理相关页面。系统基于组件化思想对页面结构进行划分，并通过响应式状态管理完成商品数据渲染、表单交互与提示信息更新。例如，当用户提交画像问卷后，系统可以立即刷新推荐区域；当用户登录状态变化时，导航栏和受保护页面也会根据权限自动调整展示内容。这些都体现了 Vue3 在动态界面构建中的优势。

### 2.2.3 Vue Router 路由管理

Vue Router 是 Vue 生态中的官方路由管理工具，用于构建单页应用中的页面跳转逻辑。传统多页应用在页面切换时通常需要重新加载整个文档，而单页应用通过路由控制视图切换，可以在保持页面整体不刷新的情况下完成内容更新，从而获得更流畅的用户体验。

本系统前端页面数量较多，既包含用户端页面，也包含后台管理页面，因此需要一个清晰的路由组织方案。Vue Router 在本系统中主要承担三个方面的作用。第一，用于定义不同业务页面的访问路径，例如首页、商品详情、购物车、订单中心、画像问卷和后台商品管理等页面均通过路由进行组织；第二，用于实现页面参数传递和动态跳转，例如商品详情页根据商品编号动态展示对应内容；第三，用于实现登录状态校验与角色权限控制，通过路由守卫机制限制未登录用户访问受保护页面，并限制普通用户访问后台管理路径。

通过 Vue Router 的引入，系统前端导航结构更加清晰，页面切换效率更高，同时也为权限控制提供了用户体验层面的第一道防线。

### 2.2.4 Axios 异步通信机制

Axios 是一个基于 Promise 的 HTTP 客户端库，常用于浏览器端与后端接口之间的数据通信。相较于原生 Ajax 实现，Axios 在请求封装、响应拦截、错误处理和统一配置方面更加便捷，因此被广泛应用于前后端分离项目开发中。

在本系统中，Axios 主要承担前端与后端 REST 接口的数据交换任务。用户登录后，前端需要通过 Axios 调用认证接口获取 JWT；在商品列表页中，需要调用商品查询接口获取商品数据；在画像问卷页中，需要调用画像保存接口提交用户填写内容；在购物车和订单页中，则需要与购物车接口和订单接口持续交互。为了提高前端代码可维护性，本系统对 Axios 进行了统一封装，通过请求拦截器自动携带令牌信息，通过响应拦截器统一处理权限失效和异常提示，从而避免在每个页面重复编写认证与异常逻辑。

此外，Axios 的异步通信模式也使页面能够实现局部刷新。例如，用户点击“加入购物车”后，无需重新加载整个页面即可提示操作结果；后台管理员修改订单状态后，也可以快速刷新订单列表。这种方式提升了系统交互的即时性和整体使用体验。

### 2.2.5 Element Plus 组件库

Element Plus 是基于 Vue3 的桌面端组件库，提供了表单、表格、弹窗、菜单、按钮、标签、分页和消息提示等常用 UI 组件，适合中后台管理系统和业务型前端项目开发。其优点在于组件丰富、文档完善、开发效率高，能够显著减少基础界面代码编写工作量。

本系统在用户端与管理端均使用了 Element Plus 提供的组件能力。在用户端，登录注册表单、问卷选项、消息提示和对话框等功能依赖该组件库完成，从而提升了交互一致性与界面规范性；在管理端，商品列表表格、订单数据展示、状态开关和编辑表单等也借助 Element Plus 快速构建。由于后台系统涉及较多数据录入和表格操作，Element Plus 在表单校验、表格展示和按钮操作上的成熟能力对提升开发效率具有重要作用。

### 2.2.6 Vite 工程化构建工具

Vite 是新一代前端构建工具，具备启动速度快、热更新效率高和打包配置简洁等特点。传统前端工程构建工具在开发环境下常存在冷启动慢、改动后编译时间长等问题，而 Vite 通过按需编译和现代浏览器原生模块支持，有效提升了开发体验。

在本系统实现过程中，Vite 主要用于前端项目创建、开发环境运行和生产构建。其快速热更新能力使页面布局调整、组件样式修改和交互逻辑调试能够更及时地反馈到浏览器界面中，适合毕业设计项目中高频率的功能迭代需求。同时，Vite 对 Vue3 的支持较为完善，能够与组件化工程结构良好结合，从而形成较稳定的前端开发环境。

### 2.3 后端技术体系

### 后端部分承担系统的核心业务逻辑处理与数据接口提供，是整个系统运行的关键支撑。本系统后端采用 Spring Boot 3.2.5 作为开发基础，并结合 Spring Web、Spring Security、JWT、Spring Data JPA 和 Bean Validation 等技术实现接口服务、安全控制和数据访问管理。

### 2.3.1 Spring Boot 框架

### Spring Boot 是基于 Spring 生态构建的快速开发框架，其主要优势在于“约定优于配置”，能够通过自动装配机制简化传统 Java Web 项目中繁琐的环境配置过程。开发者只需引入相关依赖并进行少量配置，即可快速构建结构清晰、可扩展性良好的后端应用\[15]\[16]。

### 在本系统中，Spring Boot 主要用于搭建 RESTful API 服务，负责处理用户登录注册、商品查询、购物车管理、订单提交与状态维护、画像保存以及推荐计算等核心业务。相较于传统 Servlet/JSP 开发模式，Spring Boot 更适合前后端分离系统的开发方式，其控制层、服务层与数据层的组织方式也更契合本系统多模块协同实现的需求。

### 2.3.2 Spring Web 与 RESTful 接口

### Spring Web 是 Spring Boot 中用于构建 Web 应用和接口服务的重要模块。基于该模块，开发者可以通过注解方式快速定义接口路径、请求方法和参数绑定关系，从而构建符合 REST 思想的服务接口。

### 本系统后端通过 `AuthController`、`ProductController`、`CartController`、`OrderController` 和 `UserProfileController` 等控制器类，对外提供认证、商品、购物车、订单和画像等接口。前端通过 HTTP 请求调用这些接口，以 JSON 作为数据交换格式，实现页面与业务逻辑的解耦。RESTful 风格接口具有结构清晰、调用规范统一和便于调试测试等优势，适合本系统这种前后端分离式业务场景。

### 2.3.3 Spring Security 与 JWT 鉴权机制

### Spring Security 是 Spring 生态中的安全框架，主要用于实现身份认证与权限控制。JWT（JSON Web Token）是一种轻量级令牌机制，可用于在前后端分离系统中保存用户登录状态。二者结合后，可以构建无状态、可扩展的登录认证方案。

### 在本系统中，用户注册或登录成功后，后端会生成携带用户身份信息的 JWT 返回给前端。前端在后续请求中将令牌放入 `Authorization` 请求头，由后端过滤器在请求进入控制器之前进行解析，并恢复用户身份上下文。对于普通公开接口，如登录、注册和商品列表查询，可允许匿名访问；对于购物车、订单、画像和后台管理接口，则必须要求携带合法令牌。系统同时区分 `USER` 和 `ADMIN` 两类角色，以实现普通用户与后台管理员之间的权限隔离。

### 采用 JWT 的好处在于不依赖服务端会话存储，适合前后端分离场景，能够简化认证流程并提高接口扩展性。Spring Security 则负责统一拦截和校验请求，使得权限控制逻辑更加集中和规范。

### 2.3.4 Spring Data JPA 数据访问技术

### Spring Data JPA 是对 JPA 规范的进一步封装，能够通过实体类映射数据库表，并通过 Repository 接口简化数据访问层开发。开发者无需编写大量重复的 SQL 和 JDBC 模板代码，即可实现常见的增删改查与条件查询功能。

### 本系统使用 JPA 将用户、商品、用户画像、购物车、订单和订单明细等表结构映射为 Java 实体类，并通过 Repository 接口完成数据访问。该方式在毕业设计项目中具有较高的开发效率，能够帮助开发者把主要精力放在业务逻辑本身，而非底层数据库访问细节。同时，JPA 的实体关联映射方式也有利于梳理系统中“用户—画像”“订单—订单明细”等业务关系。

### 2.3.5 Bean Validation 参数校验

### Bean Validation 是 Java 后端开发中常用的参数合法性校验机制，可以通过注解方式对请求对象中的字段进行非空、长度、格式和范围等规则限制。相比在控制器中手工编写大量 `if` 判断，注解式校验具有结构清晰、复用性强和错误信息统一等优点。

### 在本系统中，用户注册、商品维护和画像问卷提交等场景都涉及数据输入。通过引入参数校验机制，可以在请求进入核心业务逻辑之前发现非法输入，及时向前端返回明确的错误提示，减少无效请求对系统造成的影响，并提高接口健壮性。

### 2.4 数据存储与安全机制

### 2.4.1 数据存储

### MySQL 是当前应用广泛的开源关系型数据库管理系统，具备部署成本低、性能稳定、生态成熟和易于维护等优点，适用于电商系统、内容管理系统和各类业务信息管理平台开发。由于本系统的数据结构具有较强的关系性，例如用户与画像之间的一对一关系、订单与订单明细之间的一对多关系、购物车条目与用户之间的关联关系等，因此采用关系型数据库进行建模更加合适。

### 在本系统中，MySQL 主要用于保存用户账户数据、商品数据、用户画像数据、购物车数据、订单主表和订单明细数据等。通过设计 `users`、`products`、`user_profiles`、`cart_items`、`shop_orders` 和 `order_items` 等核心数据表，系统能够完整支撑“用户注册登录—商品浏览—问卷画像—加入购物车—提交订单—后台处理订单”的业务流程。关系型数据库在保证数据一致性和结构规范方面具有明显优势，也便于后续对系统进行统计分析与功能扩展。

### 后端配置中预留了 Redis 依赖与连接参数，具备后续引入缓存与会话优化的扩展空间\[17]。从当前实现看，系统以 MySQL 为主存储方案，以保证业务闭环完整；从后续演进看，也为热点数据缓存和性能优化保留了技术接口。

### 2.4.2 安全机制

### 系统在处理用户账户信息时，必须满足基本的数据安全要求。本系统采用 BCrypt 算法对用户密码进行哈希处理后再保存到数据库中，避免明文密码泄露带来的安全风险。BCrypt 具有加盐机制和较强的抗暴力破解能力，因此被广泛用于 Web 系统密码存储场景。

### 除密码保护外，本系统还通过受保护接口鉴权、角色权限控制、请求异常统一处理等方式构建基本安全防线。对于后台管理接口，系统会额外校验当前用户角色是否为管理员，以防止普通用户越权访问。对于无效令牌、未登录请求和参数异常等情况，后端统一返回标准化错误信息，提升系统接口的健壮性和安全可控性。整体而言，这一安全机制能够满足毕业设计系统在真实演示场景中的基础安全需求。

### 2.5 个性化推荐方法基础

### 推荐系统是本课题区别于普通商城系统的关键部分。常见的推荐方法包括基于协同过滤的推荐、基于内容的推荐、混合推荐、知识图谱推荐以及基于深度学习的推荐方法等\[12]\[13]\[14]。不同方法各有优缺点：协同过滤依赖较多历史行为数据，适用于用户规模较大、交互记录丰富的平台；深度学习推荐虽然表达能力强，但训练复杂、对数据规模与算力资源要求较高；基于内容和规则的推荐则更适合数据量有限且强调可解释性的应用场景。

### 2.5.1 推荐方法选型依据

### 本系统处于毕业设计项目场景，用户规模有限，缺少长期积累的历史点击、收藏和购买数据，因此如果直接采用协同过滤或深度学习推荐方案，将面临明显的冷启动问题和训练数据不足问题。同时，本系统应用场景涉及健康营养商品推荐，推荐结果不仅要“能排序”，还需要“能说明为什么推荐”，以便增强用户信任度。因此，本文更强调推荐逻辑的可解释性与可落地性，而不是单纯追求复杂模型的算法先进性。

### 基于上述考虑，本文采用规则加权评分的方法实现个性化推荐。该方法能够直接利用用户画像和商品标签进行匹配计算，无需大量历史行为数据即可运行，并能通过具体得分项解释推荐依据，较适合本系统当前阶段的研究目标。

### 2.5.2 用户画像建模思想

### 用户画像是推荐计算的输入基础。本系统通过问卷采集用户的年龄段、性别、健康目标和预算偏好等信息，并将其保存为结构化画像数据。画像中的健康目标是核心特征，例如睡眠改善、护眼、运动营养、综合营养和心血管关注等，这些目标与商品标签字段之间可以形成较直接的映射关系。相比纯行为型推荐，画像型推荐更适合本系统的冷启动场景，也更符合健康消费决策中“先识别需求，再给出建议”的业务逻辑。

### 2.5.3 规则加权评分原理

### 规则加权评分方法的核心思想，是将用户画像特征与商品属性之间的适配关系拆分为多个可量化的得分项，再通过权重组合得到最终推荐分值。在本系统中，推荐分数主要由以下几个方面构成：其一，健康目标与商品标签之间的匹配得分；其二，商品价格与用户预算偏好之间的匹配得分；其三，商品适用人群与用户年龄、性别之间的适配得分；其四，在部分场景中结合通用人群偏好与行为增强项对结果进行微调。系统最终根据综合得分对商品进行排序，并将匹配度结果返回前端显示。

### 这种实现方式有三个明显优点。第一，能够较好解决冷启动问题；第二，推荐依据可回溯、可解释，便于论文中进行说明，也有利于提升用户理解度；第三，权重设置较为灵活，后续可根据测试结果进一步调整策略。

### 2.5.4 本章小结

### 本章从系统技术路线、前端技术体系、后端技术体系、数据存储与安全机制以及推荐方法基础五个方面，对本系统的核心开发技术进行了说明。可以看出，Vue3、Spring Boot、MySQL 和 JWT 等技术共同构成了系统稳定运行的工程基础，而基于用户画像的规则加权推荐方法则构成了系统个性化服务能力的核心。上述技术体系不仅满足了本课题在功能实现上的需求，也为后续章节的需求分析、系统设计、系统实现与测试分析奠定了技术基础。

# 第三章 需求分析

## 3.1 业务需求分析

需求分析是系统设计与实现的前提。结合本课题的应用背景可以发现，健康营养品销售并不是单纯的商品展示问题，而是同时涉及用户健康诉求识别、商品适配分析、交易流程组织以及后台运营维护等多个方面。若系统仅提供普通商品列表和关键词搜索功能，虽然能够满足最基本的电商购买行为，但很难回应“不同用户应该买什么”“为什么推荐该商品”“是否适合当前预算和诉求”等关键问题。因此，本系统在业务层面并非传统商城系统的简单复刻，而是在标准商城流程基础上引入用户画像和个性化推荐能力。

从业务角色来看，系统主要面向两类用户：

1. 普通用户：关注商品是否适合自身健康诉求、预算是否合理、下单流程是否便捷。
2. 管理员用户：关注商品数据维护是否高效、订单处理是否直观、推荐基础数据是否可管理。

围绕上述角色，系统需要解决以下核心业务问题：

1. 如何通过较低的交互成本获取用户基本健康偏好信息。
2. 如何将用户画像与商品标签、价格、适用人群建立对应关系，形成可解释推荐结果。
3. 如何在推荐之外仍保留完整的电商基础能力，包括商品浏览、加购、下单、订单查询等。
4. 如何为后台运营人员提供相对清晰的商品维护与订单流转管理能力。

综合分析后，本文将系统业务目标概括为“以用户画像驱动推荐，以标准交易流程承接购买，以后台管理保障数据维护”。这一目标既体现了系统的电商属性，也体现了本课题在个性化健康消费场景中的研究重点。

## 3.2 功能需求分析

### 3.2.1 用户端功能需求

用户端是系统的主要服务对象，其需求具有“路径完整、操作简洁、推荐可理解”的特点。结合实际使用流程，可将用户端功能需求划分为如下几类。

首先是账户与身份需求。用户需要能够完成注册、登录和身份保持，系统还应支持在登录后获取用户基本资料，为后续的推荐服务、订单归属和个人信息维护提供基础。由于推荐接口和订单接口涉及用户敏感业务数据，因此身份识别必须准确、稳定。

其次是商品浏览与发现需求。用户进入系统后需要能够快速浏览商品，系统应支持首页展示、搜索、分类筛选、价格区间筛选和商品详情查看等功能。对健康营养品而言，商品展示不仅仅是价格与图片的呈现，还需要向用户传达标签信息、适用场景以及与其健康诉求之间的匹配关系。

再次是个性化推荐需求。用户希望系统在“我是谁”和“我需要什么”的基础上给出差异化商品列表，而不是向所有用户展示完全相同的商品顺序。因此，系统必须提供画像问卷采集、画像结果存储、推荐列表获取及推荐匹配度展示等能力。该部分是本系统区别于普通商城系统的关键功能。

此外，交易相关需求也必须完整支持。用户在浏览并选择商品后，应能够将商品加入购物车、调整数量、删除不再需要的条目、查看价格汇总并完成订单提交。提交订单后，用户还需要查看订单状态、配送状态以及订单商品明细，并在符合业务规则的前提下取消未确认订单。

最后，用户还存在个人信息管理需求。由于订单提交依赖地址信息，系统应允许用户在个人中心中维护收货地址，并查看已保存的画像信息，从而形成较为完整的个人档案页。

综合而言，用户端功能需求可归纳为：

1. 注册、登录、退出和身份识别。
2. 首页浏览、搜索、筛选和商品详情查看。
3. 画像问卷填写、画像保存与画像结果查看。
4. 个性化推荐列表获取和匹配度展示。
5. 购物车管理、订单提交、订单查询与订单取消。
6. 个人中心地址维护与基础信息展示。

### 3.2.2 管理端功能需求

管理端承担系统基础数据维护与业务运营支撑职能。由于推荐算法依赖商品标签与适用人群字段，因此后台不仅要具备常规商品增删改查能力，还要能够为管理员提供足够清晰的数据录入参考，避免因标签书写不统一而导致推荐结果失真。

管理端的核心功能需求主要包括：

1. 商品管理需求：能够新增商品、编辑商品、删除商品，并维护价格、描述、主图、多图、标签和适用人群等字段。
2. 推荐基础数据维护需求：系统需要展示 `tags` 与 `targetAudience` 的对照说明，帮助管理员理解这些字段与推荐逻辑之间的关系，提高录入规范性。
3. 订单管理需求：管理员需查看系统全部订单，对订单确认状态和发货状态进行更新，从而支撑订单履约流程。

与用户端相比，管理端功能数量较少，但对数据一致性与操作准确性要求更高，因为其录入结果会直接影响前台展示效果与推荐质量。

### 3.2.3 推荐与风险提示需求

推荐功能是本课题的重要研究对象，因此有必要单独进行需求提炼。首先，推荐结果必须具有差异化，即不同画像用户在相同商品集合上应获得不同排序结果。其次，推荐逻辑必须具有可解释性，至少能够通过“匹配度”或相关提示让用户理解系统推荐并非随机排序。再次，推荐不能脱离健康消费场景的基本安全要求，因此在购物车阶段需要加入基础风险提示，提醒用户在高金额或组合购买时保持理性。

基于此，本系统对推荐与风险提示提出如下需求：

1. 推荐结果应综合考虑健康目标、预算区间和目标人群适配。
2. 推荐结果应支持可视化表达，如匹配度百分比。
3. 推荐接口需兼容“已建立画像用户”和“未建立画像用户”两种情况。
4. 购物车应具备基础风险提醒能力，辅助用户进行审慎下单。

## 3.3 非功能需求分析

除功能需求外，系统还必须满足一定的非功能指标，这些指标直接影响系统使用体验和后续维护成本。

### 3.3.1 性能需求

本课题属于毕业设计系统，部署环境和用户规模均相对有限，但仍需保证在正常使用场景下页面加载流畅、接口响应稳定。具体而言，首页商品查询、推荐列表获取、购物车查询和订单查询等高频接口应具有较好的响应速度，避免影响用户连续操作体验。

### 3.3.2 安全需求

由于系统涉及用户账户、地址信息和订单数据，因此必须具备基本安全能力。密码不能明文存储；受保护接口必须经过身份校验；后台功能必须与普通用户功能隔离；跨域访问也应在可控范围内完成配置。这些要求与现代 Web 系统基本安全规范一致，也是采用 Spring Security 与 JWT 的主要原因。

### 3.3.3 可靠性需求

系统应确保核心业务流程可持续执行，尤其是“登录—获取用户信息—提交画像—获取推荐—加入购物车—提交订单”这一主链路不能出现大面积中断。对于输入异常、未登录访问、角色不匹配等情况，系统应返回明确提示，避免用户无法判断错误原因。

### 3.3.4 易用性需求

健康营养品用户的专业背景差异较大，因此系统界面需要尽量简洁易懂。问卷填写过程不宜过于复杂，推荐展示不宜过于抽象，后台数据维护也应有明确提示说明。良好的易用性能够降低学习成本并提升系统整体完成度。

### 3.3.5 可维护性与可扩展性需求

考虑到后续可能引入真实大模型问答、复杂推荐算法、支付闭环和缓存机制，系统结构必须具备良好可维护性。具体表现为前后端职责清晰、后端分层明确、数据库结构相对规范。已有研究指出，基于 Spring Boot 的 Web 应用开发模式在模块组织与扩展性方面具有明显优势\[15]\[16]，这也为本系统后续演进提供了技术基础。

## 3.4 可行性分析

### 3.4.1 技术可行性

本系统所采用的 Vue3、Spring Boot、MySQL 等技术均为当前主流开发方案，具有文档完备、生态成熟、开发效率高等优点\[15]\[16]\[18]\[19]。前端技术适合快速构建交互式页面，后端技术适合搭建 REST 服务与安全控制，数据库技术适合结构化业务数据存储。因此，从实现手段看，本课题具备较强技术可行性。

# 第四章 系统设计

## 4.1 总体架构设计

本系统采用浏览器/服务器（B/S）架构，并基于前后端分离思想进行实现。前端应用负责页面展示、用户交互与状态控制，后端应用负责业务逻辑处理、接口输出与安全控制，数据库负责业务数据持久化存储。该架构能够有效降低页面展示逻辑与业务处理逻辑之间的耦合度，便于前后端并行开发，也便于后续模块扩展与部署调整\[18]\[19]。

从结构层次上看，系统可划分为以下三个层面：

1. 表现层：由 Vue3 页面、组件、路由和表单交互组成，负责向用户呈现业务入口。
2. 业务层：由 Spring Boot 控制器、服务类和安全配置组成，负责处理请求、执行业务规则并返回结果。
3. 数据层：由 MySQL 数据表和 JPA 仓储接口组成，负责完成持久化操作。

在这一总体架构之下，系统既能够满足普通用户的前台购物与推荐需求，也能够支持管理员的后台维护工作。建议在论文终稿中补充“系统总体架构图”，以直观展示前端、后端和数据库之间的关系。

## 4.2 业务流程设计

### 4.2.1 用户端业务流程

用户端业务流程从注册登录开始，到个性化推荐和订单完成结束，是系统的主要业务主线。其具体过程如下：

1. 用户进入系统后可先浏览商品，也可注册并登录账号。
2. 登录后，用户可填写画像问卷，系统保存年龄段、性别、健康目标和预算等信息。
3. 首页根据用户画像调用推荐接口，对商品进行匹配排序后展示。
4. 用户查看商品详情，并根据需要加入购物车。
5. 用户在购物车中修改数量、查看价格汇总与风险提示。
6. 用户提交订单后，系统生成订单主表与明细数据。
7. 用户在“我的订单”页面查看订单状态，并对未确认订单执行取消操作。

该流程体现了“画像输入—推荐输出—购买决策—订单生成”的闭环特征，建议在论文中配套“用户端业务流程图”。

### 4.2.2 管理端业务流程

管理端业务流程相对简洁，但对系统运行稳定性具有支撑作用。管理员登录后，首先可以进入商品管理页维护商品信息，包括价格、描述、图片、标签和适用人群。随后，管理员可进入订单管理页查看系统全部订单，并根据业务情况更新确认状态与发货状态。通过这一流程，后台用户能够持续维护推荐基础数据和订单履约数据，保障前台功能正常运行。建议在论文中补充“管理端业务流程图”。

## 4.3 功能模块设计

根据系统需求与业务流程，本文将系统划分为六个功能模块。

### 4.3.1 认证与用户模块

该模块负责完成注册、登录、身份识别和地址维护，是整个系统的基础模块。用户只有在合法登录后，才能访问购物车、订单、画像和个人中心等受保护功能。该模块还承担获取用户角色、用户编号等信息的职责，为推荐接口和后台权限控制提供基础数据。

### 4.3.2 商品展示模块

商品展示模块负责商品列表展示、搜索、分类筛选、价格筛选和详情展示。系统在首页中对商品进行网格化展示，并提供标签筛选与价格区间筛选能力；在详情页中则突出商品图片、描述、价格、标签及推荐商品，从而为用户提供更完整的购买依据。

### 4.3.3 画像采集与推荐模块

该模块是系统的核心特色模块。画像采集部分通过问卷方式获取用户个体特征，推荐计算部分则根据用户画像和商品属性完成打分排序。设计该模块时，本文重点考虑了如下问题：

1. 问卷字段应尽量精简，避免用户填写负担过重。
2. 推荐逻辑应尽量可解释，避免形成黑箱推荐。
3. 推荐策略应兼顾冷启动场景，在无历史行为数据时仍可运行。

基于此，本文选择年龄段、性别、健康目标和预算四个画像维度，与商品标签和适用人群字段形成映射关系。其中健康目标在当前项目中主要对应睡眠改善、护眼、运动营养、综合营养和心血管关注等类别，能够与商品标签形成直接匹配。

### 4.3.4 交易模块

交易模块包括购物车管理、订单生成和订单状态查询三个部分。购物车负责保存用户当前待购买商品；订单模块负责将购物车条目固化为订单数据；订单查询模块负责向用户和管理员展示订单结果与状态变更情况。该模块要求数据一致性较高，因此在设计中采用“先生成订单主记录，再生成明细记录，最后清空购物车”的顺序逻辑。

### 4.3.5 后台管理模块

后台管理模块包括商品管理和订单管理两部分。由于推荐质量依赖商品标签与适用人群填写的规范性，后台页面除提供常规编辑能力外，还展示字段对照说明，帮助管理员理解这些字段的语义与作用，降低输入歧义。

## 4.4 数据库设计

数据库设计直接决定系统的数据组织方式与后续维护效率。结合系统业务特点，本文采用关系数据库模式对核心数据进行持久化管理。

### 4.4.1 概念结构设计

从实体关系角度看，系统主要包含以下实体：

1. 用户实体 User：描述系统账号、登录凭证、角色和联系方式。
2. 用户画像实体 UserProfile：描述用户年龄段、性别、健康目标和预算偏好。
3. 商品实体 Product：描述商品名称、描述、价格、标签和适用人群。
4. 购物车实体 CartItem：描述用户购物车中某一商品及其数量。
5. 订单实体 ShopOrder：描述订单主信息，包括总金额、地址和状态。
6. 订单明细实体 OrderItem：描述订单中每个商品条目的数量、价格和小计。

其主要关系如下：

1. User 与 UserProfile 之间为一对一关系。
2. User 与 CartItem 之间可视为一对多关系。
3. ShopOrder 与 OrderItem 之间为一对多关系。

这些实体与关系能够覆盖系统核心业务流程，并保持数据结构相对简洁。

### 4.4.2 逻辑结构设计

在逻辑实现上，系统依据 `schema.sql` 设计了六张核心数据表。

`users` 表主要用于保存账号信息。字段包括用户名、密码、邮箱、手机号、角色、启用状态和收货地址等，其中用户名、邮箱和手机号均需要具备唯一性约束，以保证身份数据的准确性。

`products` 表主要用于保存商品静态信息。除名称、描述、价格、图片外，还包含 `tags` 和 `target_audience` 字段。前者用于描述商品功能标签，例如 `sleep`、`eyes`、`fitness`、`nutrition`、`cardio` 等；后者用于描述适用性别、年龄段或通用人群，这两类字段会直接参与推荐计算。

`user_profiles` 表用于保存用户问卷结果。由于健康目标可能包含多个选项，因此在当前实现中采用逗号拼接字符串形式存储，既满足当前业务需要，也便于后续进行分词和集合匹配。

`cart_items` 表用于保存购物车中的商品快照信息。选择保存商品名称与价格快照，而非仅保存商品 ID，是为了在购物车展示与订单生成过程中减少额外查询复杂度。

`shop_orders` 表用于保存订单主信息，核心字段包括用户名、收货地址、总金额、确认状态、发货状态和创建时间。

`order_items` 表用于保存订单明细信息，与 `shop_orders` 通过订单编号关联，记录商品名称、单价、数量和小计等字段。

### 4.4.3 数据库设计合理性分析

上述数据库设计具有以下优点：

1. 结构简洁，能够覆盖当前业务核心场景。
2. 主表与明细表分离，符合订单数据组织常见模式。
3. 商品标签和画像字段设计便于支持推荐算法实现。
4. 保留了未来扩展空间，例如支付状态、物流状态、行为日志等字段可在后续版本继续扩充。

## 4.5 安全与权限设计

系统在设计阶段引入了较为清晰的权限控制思路，即“前端提示性控制 + 后端强制性控制”的双层方案。

前端侧通过路由守卫对 `/cart`、`/orders`、`/profile`、`/profile-survey` 和 `/admin` 等页面进行访问判断，未登录用户会被重定向到登录页，非管理员用户则不能进入后台页面。该机制主要提升交互友好性，减少无效跳转和误操作。

后端侧通过 Spring Security 配置统一拦截请求，白名单接口允许匿名访问，其余接口需携带合法 JWT。过滤器在请求进入控制器前解析 token 并建立安全上下文，控制器再根据当前用户和角色决定是否允许访问。对于后台订单查询与状态修改等功能，系统进一步校验当前角色是否为 `ADMIN`。

这种设计方式符合现代 Web 系统安全控制常用思路，既保证了权限判断的严谨性，又兼顾了前端体验一致性。

# 第五章 系统实现

## 5.1 系统实现环境与工程组织

### 5.1.1 实现环境

系统实现环境主要包括以下组成部分：

1. 前端开发环境：Node.js、Vite、Vue3、TypeScript、Element Plus。
2. 后端开发环境：JDK 17、Spring Boot 3.2.5、Maven。
3. 数据库环境：MySQL 8.x。
4. 开发操作系统：Windows。

上述环境能够满足前后端开发、接口联调、数据库管理和项目打包运行等要求。其中，Vue3 适用于构建响应式页面界面，Spring Boot 适用于快速搭建模块清晰的 Web 服务\[15]\[16]，二者组合也是当前中小型管理系统与商城系统常见的实现方式\[18]\[19]。

### 5.1.2 工程组织

项目采用前后端分离目录结构。`frontend/` 目录保存页面、路由、组件和前端配置文件；`backend/` 目录保存控制器、服务、实体、仓储和配置类；数据库脚本通过 `create_database.sql` 与 `schema.sql` 提供初始化支持。这种工程组织方式能够将表现层与业务层明确分离，便于独立调试和维护。

从代码结构上看，后端按职责划分为 `controller`、`service`、`repo`、`entity`、`config`、`security` 等包，前端按 `views`、`components`、`router` 等目录组织页面和公共组件，符合常见工程实践。

## 5.2 用户与认证模块实现

### 5.2.1 注册功能实现

注册功能由后端 `AuthController` 提供接口支持。当前端提交用户名、密码、邮箱和手机号后，后端首先校验用户名、邮箱和手机号是否已存在，避免重复注册；随后调用 BCrypt 对密码进行加密并写入数据库。注册成功后，系统直接生成 JWT 并返回给前端，减少用户注册后的重复登录步骤。

这一实现方式具有两个优点：一是保证账号信息的唯一性与密码安全性；二是提升了注册后的连续操作体验，使用户可以在完成注册后直接进入问卷填写或商品浏览流程。

### 5.2.2 登录与身份获取实现

登录功能同样由 `AuthController` 完成。系统根据用户名查询数据库记录，再使用密码编码器比对前端输入的原始密码与数据库中的加密密码。验证通过后，系统签发以用户名为主体的 JWT，并在前端本地存储中保存 token、用户名、角色及用户 ID。

为了支撑页面级权限判断和后续推荐接口调用，前端在登录成功后继续请求 `/api/auth/me` 接口获取当前用户基础信息。这样做可以避免仅依赖前端缓存猜测用户身份，从而提高数据可靠性。

### 5.2.3 权限控制实现

认证模块的权限控制主要由三个部分协同完成：

1. `SecurityConfig`：配置匿名可访问接口与需鉴权接口。
2. `JwtAuthenticationFilter`：在请求链中解析 `Authorization` 头并恢复登录主体。
3. 前端路由守卫：限制未登录用户进入用户功能页，限制非管理员进入后台页。

该实现兼顾了后端安全性与前端交互引导，保证了系统权限体系的完整性。

## 5.3 画像与推荐模块实现

### 5.3.1 画像问卷实现

画像问卷页面由前端 `ProfileSurvey.vue` 实现。页面采用单选与多选相结合的方式，引导用户填写年龄段、性别、健康目标和预算偏好。问卷字段数量控制在较低水平，一方面避免用户填写负担过重，另一方面也便于与当前推荐算法形成直接映射关系。当前项目中的健康目标选项包括睡眠改善、护眼、健身增肌或减脂、营养补充以及心血管健康关注等内容。

提交问卷时，前端先通过 `/api/auth/me` 获取当前用户 ID，再将问卷结果提交至 `/api/user-profiles`。后端在 `UserProfileController` 中采用 upsert 方式进行处理，即如果用户画像已存在则更新，否则新增。该策略能够兼容用户后续修改画像的场景。

### 5.3.2 推荐接口实现

推荐功能由 `ProductController` 中的 `/api/products/recommend` 接口对外提供，核心逻辑封装在 `ProductRecommendationService` 中。当前实现首先读取全部商品数据和目标用户画像数据；如果未找到画像，则回退为普通商品列表返回；如果画像存在，则进入推荐打分流程。

这种实现方式考虑到了系统冷启动问题，即使新用户尚未填写问卷，也不会因缺失画像而导致首页无法展示商品。

### 5.3.3 推荐算法实现细节

推荐算法采用规则加权评分思想，将用户画像与商品属性之间的匹配程度量化为分值。具体计算中，系统分别计算：

1. 健康目标与商品标签之间的匹配得分。
2. 商品价格与预算偏好之间的匹配得分。
3. 商品适用人群与用户年龄、性别之间的匹配得分。
4. 根据性别、年龄组增加适度偏好增强项。

之后系统按照预设权重将上述得分进行加权组合，得到每个商品的总匹配度，并按降序返回。与纯协同过滤或深度学习推荐不同，这一方式更适合当前数据规模，也更便于论文中解释推荐依据\[12]\[13]\[14]。

### 5.3.4 推荐结果展示实现

前端首页和商品详情页均可调用推荐接口。首页在加载时会优先尝试获取当前用户 ID；若成功，则请求推荐列表，否则请求普通商品列表。商品卡片组件会将后端返回的 `relevanceScore` 转换为百分比样式显示，让用户能够更直观地理解推荐强弱。

这一实现既强化了个性化推荐的用户感知，也提高了推荐结果的可解释性和可信度。

## 5.4 购物车与订单模块实现

### 5.4.1 购物车功能实现

购物车由 `CartController` 负责。系统提供购物车查询、加入购物车、修改数量、删除条目等接口。在加入购物车时，系统根据当前用户和商品编号检查该商品是否已在购物车中，若已存在则累加数量，否则创建新条目。通过这种方式，避免了同一商品在购物车中重复生成多条记录。

购物车页面会实时计算商品总额、运费和应付金额，并根据当前购物车内容请求风险评估接口。当前实现中，当购物车总金额超过阈值时，系统会给出基础提醒信息。虽然风险评估逻辑仍然较为基础，但已经形成了“选择商品—查看汇总—收到提醒—再决定下单”的交互闭环。

### 5.4.2 订单生成实现

订单生成逻辑由 `OrderController` 中的 `/api/orders/submit-from-cart` 完成。系统在下单时首先读取当前用户的购物车数据，若购物车为空则直接返回错误提示；若存在购物车条目，则先计算订单总价，再创建订单主记录，随后根据购物车条目逐项生成订单明细，最后清空购物车数据。

这种实现方式能够保证订单数据结构清晰，也符合典型商城系统中“订单主表 + 订单明细表”的设计方式。由于订单明细保存了商品名称、单价和数量等快照信息，因此即便后续商品信息被修改，历史订单仍能保留提交时的关键数据。

### 5.4.3 订单查询与取消实现

订单模块同时为普通用户和管理员提供查询能力。普通用户可通过 `/api/orders/my` 查询本人的订单列表，系统按创建时间倒序返回订单信息及明细；管理员则通过 `/api/orders/admin` 查看全量订单。

对于用户取消订单场景，系统允许用户取消尚未确认的订单。后端在处理取消请求时会先校验订单归属是否匹配当前用户，再判断订单是否已经被确认，只有满足业务规则时才删除订单及其明细。这一实现既满足了用户自主取消需求，也避免了对已进入处理流程订单的不当修改。

## 5.5 后台管理模块实现

### 5.5.1 商品管理实现

后台商品管理页通过 `/api/products` 接口获取商品列表，并在页面中展示商品编号、名称、标签、适用人群、价格、主图和描述等信息。管理员可在该页面执行新增、编辑和删除操作。

与一般商品后台不同的是，当前实现额外展示了 tags 对照表和 targetAudience 对照表，用于提示字段语义。例如 `sleep` 对应睡眠改善，`cardio` 对应心血管支持，`female` 和 `31-50` 等 token 分别对应性别和年龄段。这种设计能够直接服务于推荐逻辑，提高后台录入的规范程度。

### 5.5.2 商品编辑实现

商品编辑页支持创建新商品和修改已有商品。管理员可录入名称、价格、描述、主图 URL、图片列表、标签和适用人群等字段。前端支持图片预览与主图设置，提交时会将图片列表序列化后传给后端保存。后端在 `ProductController` 中统一处理商品新增与修改逻辑，并对输入文本做去空白与空值规范化处理。

该实现既满足了日常商品维护需求，也为推荐模块提供了结构化数据来源。

### 5.5.3 订单管理实现

后台订单管理页通过 `/api/orders/admin` 获取全部订单，并采用开关组件让管理员直接更新订单的确认状态和发货状态。后端在更新状态时会再次校验当前用户是否为管理员，确保该操作不会被普通用户调用。

订单管理页面展示订单编号、用户名、创建时间、地址、明细列表和总金额等信息，方便管理员快速掌握订单处理进度。管理员可分别控制“已确认”和“已发货”两类状态，使订单流转过程更贴合当前项目的实现逻辑。通过这一模块，系统形成了前台下单与后台履约之间的完整连接。

# 第七章 总结

本文围绕“基于用户画像的个性化健康营养商城系统”完成了从需求分析、系统设计、功能实现到测试验证的全流程工作。系统以 Vue3 和 Spring Boot 为核心技术，构建了用户画像驱动的推荐机制和完整交易闭环，实现了较好的工程可落地性与展示效果。

本文主要成果包括：

1. 构建了前后端分离的健康营养商城系统架构。
2. 实现了用户画像采集与规则加权推荐方法。
3. 打通了购物车、订单和后台管理的业务闭环。
4. 在测试中验证了系统功能正确性和基础稳定性。

系统仍存在一定不足：

1. 推荐算法尚未引入用户行为反馈学习。
2. 智能问答能力与知识约束机制仍需完善。
3. 支付、安全审计与自动化测试体系有待加强。

后续可从以下方向持续优化：

1. 引入混合推荐与在线学习机制，提高长期推荐效果。
2. 接入可控大模型问答框架，增强专业解释能力。
3. 完善支付链路与风控机制，提升工程完备性。
4. 建立自动化测试和性能压测流程，提高系统鲁棒性。
